# Phantom Ledger — ML System Design (Fraud Detection)

## 1. Problem Statement

Given a transaction at authorization time, predict the probability it will result in a fraud chargeback within 60 days. Output a score in `[0,1]` and a decision in `{approve, decline, step_up_3ds}` under tight latency. Class is highly imbalanced (~0.5% positive). Labels arrive late and adversaries adapt.

Success criteria:
- **Modelling**: PR-AUC ≥ 0.55 on a stationary holdout; precision@1% ≥ 0.40.
- **Serving**: scoring p99 < 100 ms, throughput 2k QPS sustained.
- **Operational**: drift detection within 24 h; retrain + safe rollout within 72 h.

## 2. High-Level Design

```
                           ┌────────────────────┐
        tx.events ────────►│ stream-processor   │── upsert ─► Redis (online features)
                           └────────────────────┘
                                                          ▲
        ┌──────────┐  POST /score   ┌─────────────────┐   │ HGET
        │ acquirer │◄──────────────►│  scoring-api    │───┘
        └──────────┘                │ (FastAPI+LGBM)  │
                                    └────────┬────────┘
                                             │ decision log
                                             ▼
                              gs://…/decisions/         (Parquet, hourly)
                                             │
                                             ▼
              ┌─────────────────────────────────────────┐
              │  monitoring jobs (CronJob, daily)       │
              │  - PSI / KS  - calibration  - PR-AUC    │
              └────────────┬────────────────────────────┘
                           │ metrics + alerts
                           ▼
              ┌─────────────────────────────────────────┐
              │  retraining flow (Prefect)              │
              │  build set → train → eval → register    │
              │  → shadow → canary → prod               │
              └─────────────────────────────────────────┘
                           │
                           ▼
                    MLflow Registry  (alias: production / shadow)
```

Components:
- **Online inference**: FastAPI service, LightGBM in-process; reads features from Redis; logs every decision.
- **Feature service**: today, embedded in scoring-api (single Redis call); leaves room to extract later.
- **Training pipeline**: Prefect flow, runs on K8s as Job, reads gold from GCS.
- **Model registry**: MLflow with Postgres backend + GCS artifacts; aliases `production`, `shadow`, `canary`.
- **Monitoring**: separate CronJob that produces drift/calibration metrics and triggers retraining.

## 3. Low-Level Design

### 3.1 Label Contract

Two label sources co-exist:

**A. Production (steady state)** — chargeback feedback from issuers.
- `label = 1` if any chargeback with `reason_group ∈ {fraud_cnp, fraud_ato, fraud_lost_stolen}` is linked to the transaction within 60 days of `event_timestamp`.
- **Stability**: a transaction's label is considered stable only if `today - event_date ≥ cutoff_days` where `cutoff_days = max(60, P95(cb_delay))` (dynamic per Section 03 S4).
- **Negatives**: all transactions older than `cutoff_days` without a fraud chargeback. Refunds, friendly fraud (`reason_group = dispute_quality`) are excluded — neither positive nor negative; they go to a `holdout` bucket.
- **Storage**: materialized in `gold.ml_training_set/cutoff_date=…`; one row per `tx_id` snapshotted with its features at `event_timestamp` (point-in-time).

**B. Bootstrap** — IEEE-CIS Fraud Detection public dataset (`isFraud` column).
- Used for the initial model (cold start, no chargeback history yet).
- IEEE-CIS labels are **transaction-level user-flag**: if a user is flagged fraud, *all* their transactions are labeled positive — known label noise; documented in the model card.
- Mapping IEEE-CIS columns to Phantom Ledger schema is fixed in `RENAME` (see [docs/train_flow.md §2.1](../docs/train_flow.md)).
- Once production labels accumulate to ~6 months, the model is retrained on production labels and the bootstrap source is retired.

### 3.2 Split Policy

Temporal, with gap:

| Split | Window | Purpose |
|---|---|---|
| Train | `[cutoff − 180d, cutoff − 60d − 30d]` | model fitting |
| Validation | `[cutoff − 90d, cutoff − 60d]` | hyperparam, threshold |
| Test | `[cutoff − 60d − 7d, cutoff − 60d]` | reported metric |

- No random split (would leak temporal structure).
- Gap of 30 days between train end and val start absorbs label-delay noise.
- For drift evaluation we additionally hold out the **future** 7 days (un-labeled at score-time, scored shadow-mode and labeled later).

### 3.3 Features

After feature engineering the candidate set is ~490 columns; a top-K by gain selection trims this to **250** (`TOP_K_FEATURES`) before training. Six families:

1. **Transaction-level**: `amount_usd`, `amount_log`, `amount_cents`, `mcc`, `channel`, `is_late`, `hour_of_day`, `day_of_week`, `is_weekend`.

2. **Card rolling** (from `feat_card_rolling` / Redis): tx_count and amount aggregates over 1m/5m/1h/24h/7d/30d, distinct merchants, distinct countries, velocity p50/p95, `card_tx_count_so_far`, `card_amount_mean_so_far`, `amount_zscore_card`.

3. **Merchant rolling**: chargeback rate 30d, tx volume, fraud-prior risk_tier.

4. **Identity** (from joined identity table): `device_brand` (parsed from `device_info`), `os_family` + `os_version` (parsed from `os_raw`), `browser_family` + `browser_version` (parsed from `browser_raw`), `screen_width` / `screen_height` / `screen_area` (parsed from `screen_resolution`), `device_type`. Coverage ~25% of transactions; missing → NaN (LightGBM handles natively).

5. **User identification (UID magic)** — the highest-signal family. Four nested uids built per row, then group-aggregated:

   | UID | Composition | Intent |
   |---|---|---|
   | `uid1` | `card_id + billing_zone` | basic card-region key |
   | `uid2` | `uid1 + (day − card_age_days)` | anchors to first-seen day; stable across train/test gap |
   | `uid3` | `uid2 + email_purchaser` | separates same-cohort users by email |
   | `uid4` | `uid3 + device_brand` | same person on same device cluster |

   For each uid × `target ∈ {amount_usd, C13, D15, D4}` we compute group `mean` and `std` (~32 features). Plus `<uid>_freq` for each.

   Why the `(day − card_age_days)` anchor instead of `D3`: D3 (days since previous tx of same card) is broken by data gaps; the anchor uses an absolute reference point and stays stable.

6. **Cross / interaction**: `amount_zscore_card`, `country_change_flag`, `device_age_days`, `merchant_new_to_card_flag`.

**Email cleanup before uid construction**: `anonymous.com` and `mail.com` are replaced with NaN (placeholders carrying no identification signal; would otherwise lump unrelated users into one uid bucket).

All features are computed once in gold and snapshotted; serving and training read identical schemas via `feature_schema.json`. **No feature is fitted inside the model service** — eliminates train/serve skew. The full feature list is canonicalized in `outputs/feature_schema.json#feature_columns` and serves as the contract between training and `scoring-api`.

See [docs/train_flow.md §2.3](../docs/train_flow.md) for the implementation walk-through.

### 3.4 Models

- **Baseline**: `sklearn.LogisticRegression` with one-hot on categoricals, target encoder for high-cardinality. Sanity floor.
- **Champion**: `LightGBM`, ~1k boosting rounds, `scale_pos_weight = (#neg / #pos)`, monotonic constraint on `merchant_chargeback_rate_30d` (must increase risk).
- **Challenger (optional)**: small TabTransformer or XGBoost ensemble. Used in shadow only until proven.

Why LightGBM: handles missing natively, high-card categoricals via `categorical_feature`, fast inference (< 1ms per row), interpretable via SHAP for compliance asks.

### 3.5 Loss, Metrics, Threshold

- Train objective: binary log-loss with `scale_pos_weight`. Tried focal loss in ablation; gain marginal, skipped.
- Selection metric: **PR-AUC** (insensitive to majority class).
- Decision metric: **precision@k** with k = top 1% scored transactions per day (matches Risk Ops review capacity).
- Threshold tuning on validation:
  - `decline` if score ≥ τ_high (chosen so estimated false-positive rate on legit ≤ 0.5%).
  - `step_up_3ds` if `τ_low ≤ score < τ_high` (chosen so 3DS volume ≤ 8% of traffic).
  - `approve` otherwise.
- Calibration: Platt scaling on validation; report Brier score and reliability diagram.

### 3.6 Training Pipeline

`ml/training/pipeline.py` (orchestrated by Prefect). Concrete reference implementation: [`fraud_lgbm.py`](../fraud_lgbm.py).

```
1. resolve cutoff_date (dynamic, Section 03 S4 logic)
2. build_dataset(cutoff_date) → gold.ml_training_set/cutoff_date=…
3. concat training + scoring populations for FE so freq tables and uid groups
   see the full distribution (NOT the labels — test labels stay hidden)
4. build_features: time, amount, identity parsers, email cleanup, uid1..uid4,
   uid group aggregations, point-in-time rolling stats
5. freq_encode + encode_categoricals (mappings persisted to feature_schema.json)
6. select_features: one fast LightGBM pass, rank by gain, keep top 250
7. PHASE A — fit on first (1 - VAL_FRAC) chronologically, early-stop on the
   newest VAL_FRAC slice. Returns best_iteration + val metrics.
8. PHASE B — refit on the FULL labeled set (no holdout) for
   best_iteration × FINAL_ROUND_BUMP rounds (default 1.10). This is the
   booster that ships.
9. evaluate Phase A val metrics; log Phase B round count
10. log to MLflow run: params, metrics, model.txt, feature_schema.json,
    feature_importance.csv, SHAP summary plot
11. register model version; assign alias = `candidate`
12. run gating checks:
      - PR-AUC val ≥ champion's prior PR-AUC − 0.02
      - precision@1% val ≥ champion's − 0.02
      - calibration Brier ≤ 0.05
    pass → alias = `shadow` ; fail → alert + stop
```

**Why two phases.** Early stopping needs a validation slice that resembles the production / Kaggle test window — i.e. *future* relative to training. Phase A puts val at the tail of the timeline so `best_iteration` generalises forward; Phase B then refits on **all** labeled rows (including that val slice) with the same round count. Discarding the most recent ~15% of labels in the shipping model is needless waste. See [docs/train_flow.md §2.7–2.9](../docs/train_flow.md) for the implementation.

**Why feature selection.** The candidate set has ~490 columns after FE; many V*-style aggregates are mutually redundant. Trimming to top 250 by gain stabilises Phase A early stopping and reduces overfit (~+0.005 public LB in our IEEE-CIS bootstrap).

Cadence:
- **Weekly** scheduled retrain on rolling 6-month window.
- **Triggered** retrain when monitoring job emits `retrain_required=true` (see §3.8).

### 3.7 Inference Path

`services/scoring-api/`:

- FastAPI with one endpoint `POST /score`.
- On startup: loads `model.txt` (LightGBM booster) and `feature_schema.json` (the contract from training: rename map, freq tables, categorical encoders, feature column order, training reference timestamp).
- Pulls model URI by alias `production` from MLflow; pins version. Periodic poll (every 60s); if `production` alias points to a new version, hot-swap behind a lock.
- Per request:
  1. Validate Pydantic request (`tx_id`, `account_id`, `merchant_id`, raw fields).
  2. Apply `schema.rename_at_ingest` to align legacy field names.
  3. Pipelined `HMGET` from Redis: `feat:account:{account_id}:pre@{tx_id}` (snapshot just before this tx, written by stream-processor), plus `feat:uid{2,3,4}:{hash}` and `feat:merchant:{merchant_id}`.
  4. In-process derived features: `hour_of_day`, `amount_log/cents`, identity parsers (`device_brand`, `os_family`, ...), uid string construction.
  5. Apply `schema.freq_tables` (unseen → 1) and `schema.categorical_encoders` (unseen → `"missing"` bucket).
  6. Assemble vector in `schema.feature_columns` order. Missing keys → `cold_start=true`.
  7. `booster.predict(vector)` → score.
  8. Apply threshold → decision.
  9. Append to in-memory ring buffer; flushed every 1s to GCS Parquet via async writer.
  10. Return `{tx_id, score, decision, model_version, cold_start}`.

The full request walk-through, latency budget, and failure modes are in [docs/inference_flow.md](../docs/inference_flow.md). A local single-row reference implementation lives in [`score_one.py`](../score_one.py); a batch counterpart in [`predict.py`](../predict.py).

**Critical invariant — zero train/serve skew**: identity parsers (`_device_brand`, `_os_family`, `_browser_family`, screen parser) and the uid recipe are imported by both `fraud_lgbm.py` (training) and `scoring-api` from a shared `phantom_features` package. A change to a parser is a single PR that affects both sides.

Performance:
- p99 < 100 ms target. Internally measured p99 < 25 ms; head-room covers network + Redis tail.
- HPA on CPU at 70%, min 3 pods, max 30.
- Pod budget: 1 vCPU, 1 GiB. LightGBM model ~10 MB.

Cold-start handling: if `cold_start=true`, decision is forced to `approve` for first transaction of brand-new account when score < τ_high; flagged for next-cycle review. (Configurable.)

### 3.8 Monitoring & Alerts

CronJob daily 02:00:
1. Read yesterday's decision log + the now-stable label slice (`event_date = today − cutoff_days`).
2. Compute:
   - Per-feature PSI vs 30-day baseline.
   - KS on score distribution vs last week.
   - Brier score on the now-labeled slice.
   - Realized precision@1%, recall, decline rate, 3DS rate.
3. Push metrics to Prometheus (Pushgateway) and write report to GCS.
4. Decision logic for `retrain_required`:
   - `True` if (any PSI > 0.3) AND (Brier deteriorated > 20%) — concept drift.
   - `True` if precision@1% drop > 25% week-over-week.
   - `False` if only covariate drift (PSI > 0.25 but calibration intact) — Section 03 S2 case.
5. If `True` → enqueue Prefect retrain flow.

Grafana dashboards: `Latency`, `Decision mix`, `Drift heatmap`, `Calibration`, `Retraining timeline`.

Alertmanager rules:
- `scoring_api_p99_seconds > 0.1 for 5m` → page.
- `scoring_api_error_rate_5xx > 0.5% for 10m` → page.
- `retrain_required = true` → notify Slack `#fraud-ml`.
- `retrain_failed` → page.

### 3.9 Rollout & Rollback

Aliases on the registry:
- `production` — active model.
- `shadow` — receives traffic, decisions logged but not enforced.
- `canary` — enforces decisions for a sampled % of traffic.

Promotion ladder for any new candidate:
1. **Shadow** for 24 h → compare score distributions and would-be decisions vs prod; gate: KL(score_shadow, score_prod) < 0.1 OR justified.
2. **Canary at 1%** for 24 h → compare realized precision@1% (proxy: manual-review queue match rate, since labels still 60d delayed) and decline rate; gate: decline rate within ±10% of prod.
3. **Canary at 10%** for 24 h.
4. **Promote to production** — atomic alias move; old version stays as `previous` for fast rollback.

Rollback:
- One command (`mlflow client set_model_version_alias production <previous>`); scoring-api hot-swaps within 60 s. No deploy.
- Triggers: page-level latency or error alert; precision@1% sudden drop > 30% in 1h.

Traffic split is implemented in scoring-api at request time (deterministic hash of `account_id` mod 100), not at the LB — keeps a single deployment and lets us reason about per-account consistency.

### 3.10 Reproducibility

- Every MLflow run logs: git SHA, dataset URI (`gold.ml_training_set/cutoff_date=…`), feature schema hash, lib versions (`pip freeze`), random seed.
- Model card auto-generated (`evidence/04_model_card_<version>.md`) with: data window, class balance, metrics, top-20 SHAP features, fairness slices (country, channel), known limitations.

## 4. Failure Handling

| Failure | Detection | Response |
|---|---|---|
| Redis down | scoring-api health probe + 5xx spike | fail-open to default-vector scoring + flag `feature_unavailable`; auto-page |
| Model load fails on new alias | startup probe / hot-swap try/except | keep prior model, alert |
| Training data gap (silver partition empty) | training pipeline pre-check | abort + alert; do not register a model trained on partial data |
| Decision log writer backed up | queue depth metric | scale up writer; if persistent, drop sampling rate (decisions still happen, audit downgraded) |
| Catastrophic drift (S3 simulated or real) | monitoring rule | force shadow → canary acceleration on candidate; if no candidate, freeze threshold to conservative preset |

## 5. Security

- Scoring API behind GKE Ingress with mTLS to acquirer.
- Authn: short-lived JWTs from acquirer (Workload Identity Federation), verified by API.
- Authz: per-acquirer rate limits + per-account fraud-list lookup permission.
- Inputs sanitized via Pydantic; reject unknown fields.
- Decision log contains pseudonymized identifiers (`account_id`, `tx_id`); never raw PAN.
- MLflow artifacts in GCS encrypted with CMEK; signed URLs only.
- SBOM (`syft`) and image scan (`trivy`) gating CI.

## 6. CI/CD

PR pipeline:
- `ruff`, `mypy`, `pytest`, contract tests against schema registry.
- Train a tiny model on a 10k-row fixture to verify pipeline plumbing (not metrics).
- Build & scan image; push to GAR with PR tag.

Main pipeline:
- Tag → Helm chart version bump → ArgoCD or `helm upgrade` to staging.
- Run smoke test (synthetic 100-tx replay, expected score range).
- Manual approval → prod (alias-only model promotion is *not* a deploy; it's a registry op).

Migration discipline:
- Feature schema bumps require a new feature-table version (`feat_card_rolling_v2`). Scoring-api supports two versions during transition. No silent feature changes.

## 7. Cost & Capacity

- 2k QPS × 100 ms p99 head-room → ~10 pods of 1 vCPU; HPA gives elasticity.
- Training: weekly LightGBM on ~30M rows finishes in ~20 min on a single 8-core node Job.
- Storage: silver ~50 GB / month; gold rolling features ~20 GB / month; decision log ~10 GB / month.
- Estimated GCP monthly cost (steady-state demo scale at 1/10 of above): < $200.
