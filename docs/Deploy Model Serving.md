# Deploy Model Serving on GKE with KServe

This guide covers the end-to-end steps to deploy a LightGBM model as an inference service on GKE using KServe, from creating GCP/GKE service accounts to verifying predictions.

> **Prerequisites**
> - GKE cluster is up and running with Workload Identity enabled (`--workload-pool`)
> - KServe, Knative Serving, Istio, and Cert-Manager are already installed
> - Model artifacts (`.bst`, `feature_schema.json`) are stored in a GCS bucket

---

## Architecture Overview

```
Client Request
    │
    ▼
Istio IngressGateway  (LoadBalancer)
    │
    ▼
Knative Serving  (routing + autoscaling)
    │
    ▼
KServe InferenceService  (fraud-detection)
    │  ┌─────────────────────────────────┐
    ├──│  Init: storage-initializer      │  ← downloads model from GCS
    │  └─────────────────────────────────┘
    │  ┌─────────────────────────────────┐
    └──│  Container: lgbserver           │  ← serves predictions
       └─────────────────────────────────┘
         ↑
         Uses K8s SA (kserve-sa) → GCP SA (kserve-sa@...) via Workload Identity
         → reads from GCS bucket
```

---

## Step 1 — Create GCP Service Account

Create a dedicated GCP Service Account for KServe to access GCS.

```bash
export PROJECT_ID="your-gcp-project-id"
export GSA_NAME="kserve-sa"

# Create the GCP Service Account
gcloud iam service-accounts create $GSA_NAME \
  --display-name="KServe Service Account" \
  --project=$PROJECT_ID
```

Grant the SA read access to GCS (to download model artifacts):

```bash
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --role="roles/storage.objectViewer" \
  --member="serviceAccount:${GSA_NAME}@${PROJECT_ID}.iam.gserviceaccount.com"
```

---

## Step 2 — Create Kubernetes Namespace

```bash
kubectl create namespace serving
```

---

## Step 3 — Bind GKE Service Account to GCP Service Account (Workload Identity)

Workload Identity allows a Kubernetes Service Account (KSA) to impersonate a GCP Service Account (GSA) without any key files.

There are **two sides** to this binding:

### 3a. Grant GCP permission (GSA side)

```bash
export PROJECT_ID="your-gcp-project-id"
export NAMESPACE="serving"
export KSA_NAME="kserve-sa"   # must match serviceAccount.name in values.yaml
export GSA_NAME="kserve-sa"

gcloud iam service-accounts add-iam-policy-binding \
  ${GSA_NAME}@${PROJECT_ID}.iam.gserviceaccount.com \
  --role="roles/iam.workloadIdentityUser" \
  --member="serviceAccount:${PROJECT_ID}.svc.id.goog[${NAMESPACE}/${KSA_NAME}]"
```

> **Format:** `serviceAccount:<project_id>.svc.id.goog[<k8s_namespace>/<k8s_sa_name>]`

### 3b. Annotate the Kubernetes Service Account (KSA side)

This is handled **automatically by the Helm chart** — the `service-account.yaml` template adds the required annotation using `serviceAccount.gcpServiceAccount` from `values.yaml`:

```yaml
# Auto-generated by Helm:
apiVersion: v1
kind: ServiceAccount
metadata:
  name: kserve-sa
  namespace: serving
  annotations:
    iam.gke.io/gcp-service-account: kserve-sa@<project-id>.iam.gserviceaccount.com
```

## Step 4 — Configure `values.yaml`

Edit [`infra/k8s/helm/serving/values.yaml`](../infra/k8s/helm/serving/values.yaml):

```yaml
modelName: fraud-detection           # Name of the InferenceService

autoscaling:
  minReplicas: 1                     # Minimum pods (set to 0 for scale-to-zero)
  maxReplicas: 5                     # Maximum pods
  scaleTarget: 10                    # Concurrent requests per pod before scaling

resources:
  limits:
    cpu: "1000m"
    memory: "512Mi"
  requests:
    cpu: "256m"
    memory: "256Mi"

serviceAccount:
  name: kserve-sa                    # K8s SA name (must match Step 3)
  gcpServiceAccount: "kserve-sa@<PROJECT_ID>.iam.gserviceaccount.com"

predictor:
  # GCS path to your MLflow artifact directory containing model.bst
  storageUri: "gs://<your-bucket>/mlflow-artifacts/<run-id>/artifacts"
```

> **Finding the `storageUri`**: In MLflow UI, go to your run → Artifacts tab → copy the GCS path.
> The directory must contain `model.bst` and optionally `feature_schema.json`.

---

## Step 5 — Install the Serving Helm Chart

```bash
helm install serving infra/k8s/helm/serving \
  --namespace serving \
  --values infra/k8s/helm/serving/values.yaml
```

To upgrade after changing `values.yaml`:

```bash
helm upgrade serving infra/k8s/helm/serving \
  --namespace serving \
  --values infra/k8s/helm/serving/values.yaml
```

## Step 6 — Verify Deployment

### Check pods

```bash
kubectl get pods -n serving
```

Expected output (after model download completes):

```
NAME                                                          READY   STATUS    RESTARTS
fraud-detection-predictor-00001-deployment-xxxxxxxxxx-xxxxx   2/2     Running   0
```

> `2/2` = `kserve-container` (lgbserver) + `queue-proxy` (Knative). The init container `storage-initializer` downloads the model and exits before the main containers start.

### Check InferenceService status

```bash
kubectl get inferenceservice -n serving
```

```
NAME              URL                                               READY
fraud-detection   http://fraud-detection-predictor.serving...       True
```

### Check storage-initializer logs (if pod is stuck)

```bash
kubectl logs <pod-name> -n serving -c storage-initializer
```

Common errors:
| Error | Cause | Fix |
|---|---|---|
| `403 iam.serviceAccounts.getAccessToken denied` | Workload Identity not bound | Re-run Step 3a |
| `GCS authentication failed` | GSA missing GCS permission | Re-run `roles/storage.objectViewer` grant |
| `No such object` | Wrong `storageUri` | Check GCS path contains `model.bst` |

## Step 7 — Test Inference

Get the Istio Ingress Gateway external IP:

```bash
export INGRESS_IP=$(kubectl get svc istio-ingressgateway \
  -n istio-system \
  -o jsonpath='{.status.loadBalancer.ingress[0].ip}')

echo "Ingress IP: $INGRESS_IP"
```

Send a prediction request (KServe v2 protocol):

```bash
curl -X POST "http://${INGRESS_IP}/v2/models/fraud-detection/infer" \
  -H "Host: fraud-detection-predictor.serving.example.com" \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": [{
      "name": "input-0",
      "shape": [1, <num_features>],
      "datatype": "FP32",
      "data": [[<feature_values>]]
    }]
  }'
```

Expected response:

```json
{
  "model_name": "fraud-detection",
  "outputs": [{
    "name": "output-0",
    "datatype": "FP32",
    "data": [0.032]
  }]
}
```

## Step 8 — Autoscaling Behavior

KServe uses Knative autoscaling based on concurrent requests:

| Setting | Value | Meaning |
|---|---|---|
| `minReplicas` | 1 | Always keep 1 pod running (no cold start) |
| `maxReplicas` | 5 | Scale up to 5 pods under load |
| `scaleTarget` | 10 | Scale up when >10 concurrent requests/pod |

To enable **scale-to-zero** (pod goes to 0 when idle), set `minReplicas: 0`. This reduces cost but adds cold-start latency (~10-30s).
